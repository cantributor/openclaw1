# 2026-06-24

## Проблема: «потерялся» контекст беседы + ошибка memory_search

### Суть
Can в webchat задал вопрос про роуминг МТС «Комфорт» в Грузии, сославшись на беседу «выше». Я её не видел в переданном контексте → показалось, что разговор утерян. Параллельно `memory_search` упал с ошибкой провайдера.

### Что выяснили

**1. Контекст НЕ утерян.**
- Сегодняшняя беседа про телефон/eSIM/роуминг/Грузию шла в той же сессии `agent:main:main` (webchat), файл `03ef9a7f-e4dd-499f-aa1b-2a12a7804df9.trajectory.jsonl`, реплики до 12:43.
- Вопрос в 14:52 пришёл в **новый прогон** той же сессии; `sessions_history` отдал только хвост с 14:52 (`truncated: true`), поэтому модель при ответе видела урезанный контекст. На диске всё цело.
- Отдельно: более ранняя беседа про связь от 16 июня — в **Telegram**-сессии `agent:main:telegram:direct:280076262` (файл `210bc284-...`). Каналы изолированы, контекст между webchat и telegram не подтягивается.
- Резюме найденной беседы: Грузия 12–28 июля; вывод — туристические eSIM только интернет; решение = симка МТС + роуминг-режим «Комфорт» (1000₽/30дн, 3ГБ+50мин, Грузия в списке), семье «Лайт» (0₽); новый телефон/eSIM для поездки не нужны; поставлены 2 напоминания (3 и 7 июля 12:00 МСК) в Telegram.

**2. Ошибка memory_search — холодный старт, не поломка.**
- `memory_search` использует эмбеддинги; провайдер `local` (плагин `llama-cpp`, модель `embeddinggemma-300m`).
- Первый вызов упал: `memory_search timed out after 15s`. В логе смежно: `startup model warmup timed out after 5000ms`. Классика холодного старта на CPU-VPS (2 ядра): первый запрос долгий (модель грузится в RAM) → таймаут. Повторные — быстрые (потом отработал за ~410мс).
- Таймаут инструмента = **15 c**, в конфиге не задан (только `agents.defaults.memorySearch = {provider: local}`). По схеме конфига (`openclaw config schema`) **прямого поля для этого таймаута нет** — зашит в код. Поэтому «поднять таймаут одной строкой» нельзя.

**3. Почему не было swap:** 
- «голый» облачный KVM-образ ставит swap без него намеренно (износ диска, философия облака). Для нас swap нужен не как рабочая память, а как подушка от OOM на безсвоповой машине — с swappiness=10 он почти не пишется. 

### Изменение направления
Сначала думали поднять таймаут (вариант 2) — отпал, поля в схеме нет. Перешли к устранению причины (холодный старт): keep-alive модели. Но keep-alive держит +0.5–0.8ГБ RAM постоянно, а на VPS **не было swap** → риск OOM (killer прибьёт gateway). Поэтому сначала сделали swap как страховку.

### Что НЕ пошло
- Вариант 2 (увеличить таймаут memory_search) — нет штатного поля в конфиге.
- `memory_search` с фильтром по словам и `sessions_list search=...` — не нашли беседу (видимость сессий ограничена `tree`; нашли через прямой разбор jsonl-файлов на диске).

### Что сделали (с результатом)

**Диагностика (на VPS, мой gateway-хост, exec без рута):**
- `free -h`, `df -h`, `swapon --show` — RAM 3.8ГБ, диск / 30ГБ свободно, swap отсутствовал.
- `systemd-detect-virt` → `kvm`, не контейнер (не OpenVZ/LXC) → swap создавать можно.
- `findmnt -no FSTYPE /` → ext4 (fallocate безопасен).
- `openclaw config schema` → выгрузил схему в /tmp/oc-schema.json, искал timeout-поля для memorySearch — прямого нет.

**Создание swap (на VPS, выполнял Can под sudo в своём терминале):**
```bash
sudo bash -c '
set -e
if swapon --show | grep -q /swapfile; then echo "swap уже активен"; exit 0; fi
fallocate -l 2G /swapfile || dd if=/dev/zero of=/swapfile bs=1M count=2048
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
grep -q "^/swapfile " /etc/fstab || echo "/swapfile none swap sw 0 0" >> /etc/fstab
swapon --show; free -h
'
```
Результат: swap 2ГБ активен, строка `/swapfile none swap sw 0 0` добавлена в `/etc/fstab`.

**Настройка swappiness (на VPS, под sudo):**
```bash
sudo sysctl vm.swappiness=10
echo 'vm.swappiness=10' | sudo tee /etc/sysctl.d/99-swappiness.conf
```
swap как аварийная подушка, не для повседневного своппинга.

**Проверка автозапуска перед ребутом (на VPS):**
```bash
systemctl --user is-enabled openclaw-gateway   # enabled
loginctl show-user "$USER" -p Linger            # Linger=yes
```
→ gateway поднимется сам после ребута без логина.

**Ребут (Can выполнил `sudo reboot`) и проверка после:**
- swap 2ГБ поднялся из fstab автоматически (`free -h`);
- swappiness=10 пережил ребут;
- gateway `active` + `enabled` — поднялся сам.

### Изменённые файлы конфигурации
- `/etc/fstab` — добавлена строка swap (на VPS).
- `/etc/sysctl.d/99-swappiness.conf` — создан, `vm.swappiness=10` (на VPS).
- `/swapfile` — создан, 2ГБ (на VPS).

### keep-alive эмбеддингов — проверено, штатно НЕВОЗМОЖНО
После swap keep-alive стал безопасен по памяти, но проверка показала, что включить его нечем:
- **JSON-схема конфига** (`openclaw config schema`): у `llama-cpp` config-опций ноль (`[]`), у `memory-core` только `dreaming`. Полей keep-alive/idle/preload/unload/ttl нет. Есть лишь общий `hooks.timeoutMs` (не про модель).
- **Исходники плагина** `@openclaw/llama-cpp-provider/dist/index.js`: grep по `keepalive|idleTimeout|unload|preload|warmup|n_keep|modelTtl|evict` → пусто. Механизма удержания/выгрузки модели по таймауту в коде нет.
- **Доки**: отдельной доки по llama-cpp keep-alive нет.
- **Текущий конфиг**: `llama-cpp = {}`, `memory-core = {}` (на дефолтах).

Вывод: вариант 3 (keep-alive) отпадает — нет штатного механизма. Костыль (внешний прогрев по cron) не делаем (против принципа «только штатным способом»). Остаётся вариант 1 — оставить как есть: память живая (~0.4с после прогрева), редкий холодный таймаут на первом запросе просто повторить. Болевой точки, требующей действий, не осталось. Swap сделан не зря — это страховка VPS от OOM, независимая от эмбеддингов.

### Принятое правило (итог работы)
Раз keep-alive штатно невозможен, а холодный таймаут memory_search остаётся — договорились жить с этим через поведенческое правило (не костыли):
- при таймауте/ошибке `memory_search` — **повторить 3 раза** перед тем, как сказать «не помню»;
- **каждую попытку показывать в чате** (Can хочет видеть процесс), не ретрай молча;
- снимает с Can ручные «вторые попытки» переспросить.
